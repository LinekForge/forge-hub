## 三个组件


### 1. Hub Server（hub-server/）

**角色**：消息路由中心 + 通道管理器。独立常驻进程。

**入口**：`hub.ts`

**核心模块**：

| 文件 | 职责 |
|------|------|
| `hub.ts` | main 入口 + onMessage 路由中枢 + WebSocket 实例注册 |
| `endpoints.ts` | HTTP + WebSocket 全部端点实现 |
| `router.ts` | 消息路由逻辑：@mention 解析、主实例 fallback、通道订阅过滤、锁定拦截 |
| `approval.ts` | 远程审批 pending state + idLookup + TTL 清理 + 持久化恢复 |
| `rate-limit.ts` | per-instance `/permission-request` 速率限制 |
| `lock.ts` | 紧急锁定（panic button）、lock-phrase 匹配 |
| `tts.ts` | TTS 合成（text → mp3 → ogg）hook 协议 |
| `asr.ts` | ASR 统一入口 `resolveAsr`：plugin `asrTranscribe` → `FORGE_HUB_ASR_HOOK` → null 三级 fallback |
| `history.ts` | 分通道聊天历史（JSONL append + 读取） |
| `hub-state.ts` | 通道 health 统计 + deriveHealthStatus |
| `channel-loader.ts` | 通道插件加载：auto-detect plugin shape，`FORGE_HUB_DEV=1` 时启用 fs.watch 热加载 |
| `channel-registry.ts` | 已加载 plugin 的 sendMap / metaMap 注册中心 |
| `channel-watchdog.ts` | 每 2 分钟扫 unhealthy 通道 stop+start 自动重启（按 `stoppedReason` 白名单） |
| `instance-manager.ts` | 实例生命周期管理：注册/注销、标签/描述持久化、在线列表 |
| `resolve.ts` | sender 昵称 → native ID 解析 + 订阅过滤 |
| `config.ts` | 路径常量、日志工具、hub-config.json 热加载、`formatUnauthorizedNotice`、`resolveFfmpeg` |
| `state.ts` | 通用持久化工具：`loadChannelState`/`saveChannelState` 读写 `state/` 子目录的 JSON |
| `types.ts` | TypeScript 接口定义（ChannelPlugin、消息格式等） |

**HTTP 端点**：

| 端点 | 方法 | 用途 |
|------|------|------|
| `/ws` | GET→WebSocket | 实例连接（升级为 WebSocket） |
| `/send` | POST | 发送文本消息 |
| `/send-file` | POST | 发送文件 |
| `/send-voice` | POST | 发送语音（Hub 层做 TTS→mp3→ogg） |
| `/health` | GET | 公开最小健康检查 |
| `/status` | GET | 详细 Hub 状态（token 模式下需鉴权） |
| `/instances` | GET | 在线实例列表 |
| `/channels` | GET | 已加载通道列表 |
| `/history` | GET | 聊天历史 |
| `/set-tag` | POST | 设置实例标签 |
| `/set-description` | POST | 设置实例描述 |
| `/set-channels` | POST | 设置实例通道订阅 |
| `/permission-request` | POST | 接收 hub-client 转发的审批请求，登记 pending、推到 `approval_channels`、返回 pushed/failed 列表 |

### 2. Hub Client（hub-client/）

**角色**：MCP channel，跑在每个 Claude Code 实例里。Hub Server 和 Claude Code 之间的桥。

**入口**：`hub-channel.ts`（单文件）

**核心逻辑**：
1. 启动时读 `next-session.json`（或 legacy `next-name.txt`）获取实例身份
2. 如果有身份信息 → 通道模式（注册为 peer，接收消息推送）
3. 如果没有 → 工具模式（只暴露工具，不连 SSE/WebSocket）
4. 通道模式下通过 WebSocket 连接 Hub Server，接收消息 → 转为 MCP notification
5. 暴露 `hub_reply` / `hub_send_file` / `hub_send_voice` / `hub_replay_history` 四个 MCP 工具
6. 处理远程审批：拦截 `permission_request` notification → POST 到 Hub → Hub 推到审批通道 → 用户回复 → Hub WebSocket 推回 → 注入 `permission_response`

---

### 远程审批数据流（详细版）

```
Claude Code 产生敏感操作
   ↓ MCP notification
   claude/channel/permission_request { request_id, tool_name, description, input_preview }
   ↓
hub-channel（MCP notification handler）
   ↓ HTTP POST /permission-request  { ...params, instance: INSTANCE_ID }
   ↓
Hub Server
   ├─ 锁定状态？→ 503
   ├─ approval_channels 为空？→ 503
   ├─ request_id 冲突？→ 409
   ├─ genDisplayIdPair() → 生成独立的 yes_id, no_id（强绑定 verdict 语义）
   ├─ 登记 pending[request_id] { yes_id, no_id, from_instance, created_at, ... }
   ├─ idLookup[yes_id] = {request_id, "allow"}; idLookup[no_id] = {request_id, "deny"}
   └─ Promise.allSettled 并发推送到 approval_channels 里所有通道
      每个通道：
      ├─ resolveApprovalRecipient(channel) → 读 state/{channel}/allowlist.json 拿 approval_owner_id
      ├─ 构造提示文案（当前 runtime 硬编码 `Claude wants to run ${tool_name}: ${description}`，[endpoints.ts:356](hub-server/endpoints.ts)）
      └─ plugin.send({ to, content, type: "text" })
      任一通道成功 → 200 { success, pushed, failed }
      全失败 → 清除 pending + idLookup → 502

用户在某一通道回复 "yes {yes_id}" / "no {no_id}"
   ↓ 通道插件轮询到（先过通道层 allowlist 第一道）
   ↓ hub.pushMessage()
Hub Server onMessage()
   ├─ [第二道防线] validateAuthorizedSender(msg)——未授权 logError+告警+丢弃
   ├─ 前置条件: pendingPermissions.size > 0（没 pending 时所有 yes/no 都是聊天）
   ├─ LOOSE 正则：/^\s*(y|yes|n|no)\s+\S/i——宽松识别"想审批"
   │   └─ LOOSE 匹配但严格不过 → malformed ack + 列正确 id（防手滑）
   ├─ STRICT 正则：/^\s*(y|yes|n|no)\s+([a-km-z]{5})\s*$/i
   ├─ getAuthSenderId(msg) + isApprovalOwner(channel, authSenderId)——只认显式 approval_owner_id
   ├─ idLookup.get(displayId) → {request_id, expected_behavior}
   │   └─ 没找到 → ack "编号不存在" + 建议 Esc
   │   └─ behavior 和 verdict 不一致 → ack "yes/id 不一致" + pending 保持
   ├─ pendingPermissions.get(request_id) → 拿到 from_instance
   ├─ instance.send({ type: "permission_response", content: JSON({request_id, behavior}) })
   ├─ send 返回 dropped/backpressure/离线？→ pending 保持，ack 提示稍后重试
   ├─ send 成功后清理: pending + idLookup[yes_id] + idLookup[no_id]
   └─ sendApprovalAck(channel, fromId, "✅ 已批准 / ❌ 已拒绝")——用户即时 confirm

hub-channel（WebSocket handleMessage）
   ↓ case "permission_response"
   ├─ 解析 + 字段校验
   └─ mcpServer.notification({ method: "notifications/claude/channel/permission", params })
   ↓ MCP notification
Claude Code 解除等待，继续执行（或拒绝）
```

**TTL 兜底**：hub-server 内存里的 `pendingPermissions` Map 每 60 秒扫一次，对超过 240 分钟（4h）未响应的条目——**先给 client 发 `behavior="deny"` 通知让 Claude Code 的 MCP handler 停下，再 delete map**。用户永远不回复的请求不会永久占内存，client 也不会挂死。

**主人识别**：审批回复识别依赖 `state/{channel}/allowlist.json` 的 `approval_owner_id`。owner 必须仍在 allowlist 中；非 owner 发的 "yes xxxxx" 会被 `isApprovalOwner` 拦住，记入日志但不转发。

**两道防线**：
1. **通道层（第一道）**：wechat/imessage/feishu/telegram 收消息时查 allowlist，未授权消息直接拒收 + push 一条 system 告警（`formatUnauthorizedNotice` 包装，抗 prompt injection）
2. **Hub onMessage（第二道）**：即使通道层漏了，onMessage 入口再做 allowlist 校验——未授权 → logError + push 告警 + **丢弃原消息**（绝不路由给 Claude Code 实例）；只有进入审批回复识别时才额外收紧到显式 owner

两道共享同一份 allowlist 文件。`fromId=system` 的消息免疫第二道 check 避免递归。

**反馈回路**：Hub 对用户每次审批回复都有 ack 回到用户发的通道（用 `sendApprovalAck`）——`✅ 已批准 / ❌ 已拒绝 / ⚠️ 不一致请重试 / ⚠️ 编号不存在建议 Esc / ⏰ 已超时`。消除"用户回复后不知道 Hub 收没收到"的黑箱。

**Pull-model history**：WS ready 时 Hub 默认只自动推 10 条 per channel（避免 CC context 被淹），agent 需要更多时主动拉：
- 默认自动推 `auto_replay_count`（default 10）条，可 config 调
- agent 需要更多时调 `hub_replay_history(channel, limit, since_ts)` MCP tool 自己拉（返回作为 tool result，不 spam 其他 instance）
- 用户保底 `fh hub replay <ch> [count]` CLI
- `GET /history?channel=X&limit=N&since_ts=Z` 是共享数据源（支持增量）

**持久化 & 观测性**：
- `state/_hub/pending.json` 持久化 pending 队列，Hub 启动时 `loadPendingFromDisk()` 恢复 + idLookup 重建
- `audit.jsonl` 4 处写审批事件（`approval_pushed/granted/denied/timeout/mismatch/push_failed/restart_timeout`）
- `fh hub pending` 看当前挂起、`fh hub approval-audit` 看历史
- `hub-config.json` 热加载（`fs.watch` + 200ms debounce），改 config 不需要 Hub 重启

**Self-test**：`fh hub self-test` spawn `hub-test-harness/harness.ts` → 后者 `mkdtemp` 700 权限目录 + spawn 独立 Hub 进程到 :9901 + 伪装 wechat 的 `test-injector.ts`（Unix socket 控制） → 跑 8 个场景（happy allow/deny、mismatch、malformed、第二道防线、无关聊天、crash × 2）→ 清理。**生产 Hub 代码里 `grep "/test"` 返回 0**——零攻击面。

### 3. CLI（forge-cli/）

**角色**：统一管理工具。单文件 TypeScript 脚本。

**入口**：`forge.ts`（通过 `~/bin/fh` symlink，`forge-hub install` 自动建立）

**工作方式**：Hub 子命令 → HTTP 调 Hub Server API（`localhost:9900`），Bearer token 鉴权可选（`HUB_API_TOKEN` env）。

**常用命令**：`fh hub status` / `peers` / `channels` / `allow` / `revoke` / `listen` / `lock` / `unlock` / `pending` / `replay` / `self-test` / `health` / `fh engine list|remove|pause|log`（完整列表见 `fh --help`）。

## 组件间通信



| 通信路径 | 协议 | 说明 |
|----------|------|------|
| Hub Client → Hub Server | WebSocket | 实例注册、消息收发、审批 |
| Hub Server → Hub Client | WebSocket | 消息推送、上下文注入、锁定通知 |
| CLI → Hub Server | HTTP | 管理操作（status / peers / send 等） |
| Hub Server → 通道平台 | 各平台 API | 微信(ilink HTTP) / TG(Bot API) / 飞书(lark-cli subprocess) / iMessage(AppleScript) |
| 通道平台 → Hub Server | 轮询 / WebSocket | 微信(HTTP轮询) / TG(长轮询) / 飞书(WebSocket订阅) / iMessage(SQLite轮询) |

## 通道插件接口



```typescript
interface ChannelPlugin {
  name: string;                        // 通道标识，如 "wechat"
  displayName?: string;                // 显示名，如 "微信"（缺省 loader 用 name fallback）
  aliases: string[];                   // 别名，如 ["wx"]
  capabilities: ChannelCapability[];   // "text" | "file" | "voice" | "image"——loader 强制过滤
  start(hub: HubAPI): Promise<void>;   // 启动轮询/监听；允许 throw（标插件加载失败）
  send(params: SendParams): Promise<SendResult>;   // 发送消息
  stop(): Promise<void>;               // 停止（loader 用 10s timeout 包住）
  isNativeId?(to: string): boolean;    // 可选：判断 to 是否已是 native ID（路由分流用）
  stoppedReason?: ChannelStopReason;   // 可选：插件自报停止原因（watchdog 决定是否自动重启）
  asrTranscribe?(audioPath: string): Promise<string | null>;  // 可选：通道自带 ASR
}
```

详细字段说明 + HubAPI 回调 + `InboundMessage` / `SendParams` / `SendResult` 见 [`hub-docs/channel-plugin-guide.md`](hub-docs/channel-plugin-guide.md) §2。

**四个已实现的通道**：

| 通道 | 消息源 | 发送方式 | Bot 身份 | 能否代 user 回第三方 | 其他 |
|------|--------|----------|----------|----------------------|------|
| wechat | iLink HTTP 轮询（含 ASR text） | iLink sendmessage API（voice = mp3 附件文件，非原生语音条） | CC 以**插件形式**寄居在 user 个人微信中 | ❌ 架构决定不行——CC 看不到 user 本人微信会话 | 需 iLink token；CDN 媒体下载偶失败；silk voice 协议不通用 |
| telegram | Bot API 长轮询（45s + 看门狗） | Bot API sendMessage | 独立 TG bot 账号 | ❌ 默认单一；**可开放**给他人用（用户按需自配 allowlist） | 代理走 env `https_proxy` / `http_proxy` |
| feishu | lark-cli event +subscribe（WebSocket） | lark-cli im +messages-send | 企业 bot（按飞书权限体系可扩展） | ❌ 默认单一；**权限体系允许扩展**到群/其他用户 | 同 app 只允许一个 WebSocket；需 `--as bot` |
| imessage | chat.db SQLite 轮询 | AppleScript | **user 本人 Apple ID**（唯一） | **✅ 唯一支持**——走本机 chat.db + AppleScript | 需 Full Disk Access；self-chat echo filter 防死循环 |

**为什么这个矩阵重要**：agent 通过 hub 工作时，如果不理解通道语义差异，可能做出架构不成立的动作——比如试图让 CC 在微信上"代 user 回朋友消息"，或假设所有通道都能开放给第三方使用。README 的"通道能力"表是面向人类的精简版，这张表是面向 agent 的完整 matrix（含消息源 / 发送方式 / 特殊要求）。
